# NPC System

Persistent, named, described non-player characters generated by Chateau interactions (currently only [Breed](../../Breed-and-Birth.md) births NPCs). NPC names and descriptions are referenced wherever the Chateau frame talks about other entities — `!work` task strings, dossier displays, achievement flavor text.

**Status:** Infrastructure spec, **low priority**. [Breed-and-Birth](../../Breed-and-Birth.md) is designed to degrade gracefully without this — births can produce un-named offspring records that this system later upgrades.

## Goal

A separate MongoDB collection of NPCs that persists across sessions, with names that look authored (not "Spawn-7842"). Once the system exists, other interactions and the `!work` system can reference NPCs in flavor text instead of using the player's own name placeholder.

## Data model

```csharp
public class NPC
{
    public string Id;                       // GUID
    public string Name;                     // user-given or generated
    public string Description;              // free text, optional
    public string Species;                  // identifier from monster catalog (or "human")
    public DateTime BornAt;
    public List<string> ParentNames;        // userNames of Chateau players involved in creation
    public string OriginInteraction;        // "breed", future: "summon", "adopt"
    public string OriginInteractionId;      // FK back to the source record (e.g. pregnancy id)
    public Dictionary<string, string> Attributes; // free-form per origin (e.g. brood-size, monster traits)
}
```

MongoDB collection: `NPCs`. Indexed on `Name` (non-unique — duplicates allowed but flagged in display).

## Naming

Per user input on 2026-05-07: large broods (insects etc.) produce many offspring but **only one is named**. The named offspring is a regular `NPC` record; the rest are aggregated as a brood-size attribute on the named one (`Attributes["brood_size"] = "47"`).

Naming options on `!birth`:

```
!birth {pregnancyIndex} name="Mothra" description="iridescent moth, hates fluorescent lights"
```

If `name=` is omitted, the system generates a name using a per-species name pool (kept in `MonDB` alongside the monster identifier). If no pool exists for that species, fall back to "{Species} Spawn #{shortId}".

## Where NPCs are referenced

Initial integrations (other specs trigger this):

- **`!work` task strings** — when the work system needs a flavor name (e.g. "Bob serves tea to {NPC}"), it draws from the player's owned NPCs (those whose `ParentNames` includes the player). If none exist, falls back to a generic name.
- **Dossier** — `!dossier` displays an "Offspring" section listing the player's NPCs.
- **Future hooks** — keep the integration thin so other interactions can opt in.

## Commands

- `!npcs` — list NPCs the caller is a parent of.
- `!npcinfo {name}` — show one NPC's record.
- `!renamempc {oldName} {newName}` — only callable by a parent. **No consent flow** since the NPC isn't a player.
- `!describempc {name} {description}` — same constraint.

## Tests

- `NPCRepositoryTests.cs`: CRUD, parent lookup, name search.
- `NPCNameGeneratorTests.cs`: per-species pool, fallback.
- `WorkNPCIntegrationTests.cs` (in the !work module if/when modified): work strings draw from NPCs when available.

## Assumptions

- NPCs do not become players. They cannot consent to or initiate interactions. They are display/flavor entities.
- One named NPC per breeding event regardless of brood size. **Override:** allow `name="A,B,C"` syntax to name N NPCs from a brood — punted unless requested.
- Renaming/describing an NPC does not require any other parent's consent. **Override:** add a multi-parent consent flow if disputes happen.
- NPC species pulls from the existing monster `Identifier` catalog. Adding human/non-monster NPC support is a future extension.
